# marimo-binder — context

## Purpose / North Star
A minimal Binder / JupyterHub-compatible repo that launches JupyterLab and auto-starts a lightweight marimo app (`marimo_app.py`) in **app mode by default**, while retaining an easy switch into **dev/editor mode** and keeping heavyweight demos optional. The entry URL should land on marimo (via a Jupyter Server root redirect to the JupyterHub proxy URL).

## Non-goals
- Not a general-purpose JupyterHub deployment repo.
- Not optimizing for complex multi-user auth, persistence, or production hardening.

## Current direction (high level)
- This repo is meant to be a Binder target; repo2docker is the fundamental deployment mechanism.
- Keep `.binder/Dockerfile` as the Binder-visible advanced repo2docker Dockerfile, and make Binder-like runtime behavior explicit inside it.
- Current `embedded-grist` work adds a pinned Grist sidecar exposed through the same JupyterHub proxy surface as marimo.
- Keep the setup reproducible (small surface area, minimal moving parts).

## Constraints / gotchas (verified)
- BinderHub variance: different BinderHub deployments may use different repo2docker versions and/or buildpack policies.
  - Observed on `binderhub.saucy.haus` (2026-05-30): it built this repo via the standard repo2docker pipeline (not our Dockerfile) and honored `.binder/start` via `R2D_ENTRYPOINT`.
  - Observed on `mybinder.org` (2026-05-30): ran the same `.binder/start` behavior (app=2718, dev=2719) for the PR head.
- Therefore: the Binder-visible Dockerfile must live at `.binder/Dockerfile`; a root-level `Dockerfile` alone is only a local/CI artifact and is ignored by repo2docker once `.binder/` is the config directory.
- Verified on `binderhub.saucy.haus` (2026-06-07): repo2docker injects Binder-style build args such as `NB_UID=1000`; the upstream `quay.io/jupyterhub/jupyterhub:5.4.6` base image already has an Ubuntu user at UID 1000, so the Dockerfile must reuse/rename that user instead of blindly `adduser --uid ${NB_UID}`.
- Git workflow: use GitButler (`but`) for mutations (staging/commits/branches).

## CI (what it guarantees)
- `docker-build`: builds `.binder/Dockerfile` (sanity that the Binder-visible image still builds).
- `docker-smoke`: runs smoke checks against the built image:
  - prints shipped versions (`marimo`, `marimo_jupyter_extension`)
  - asserts `.binder/start` actually starts marimo and embedded Grist (process / HTTP checks):
    - `marimo run marimo_app.py` (app mode, port 2718)
    - `marimo edit marimo_app.py` (dev/editor, port 2719)
    - embedded Grist (`node _build/stubs/app/server/server.js`, port 8484)
- `contract-tests`: Python unittest checks that Dockerfile/start/redirect/smoke contracts still expose Grist correctly.
- `grist-runtime-smoke`: lightweight CI gate that runs `gristlabs/grist:1.7.14` directly with the same Binder-safe defaults and verifies HTTP readiness on port 8484 without rebuilding the full Binder image.

## Repo entrypoints / behavior
- `.binder/start`:
  - SHOULD be network-free and fast (dependencies are installed at image build time)
    - Note: optional heavy demos may fetch datasets / compute at runtime, but they are not on the default Binder launch path.
  - exports `PYTHONPATH="$PWD:$PYTHONPATH"` so `marimo_redirect.py` is importable
  - starts TWO marimo instances (both `--headless --no-token`):
    - app mode (default): `marimo run marimo_app.py ... --port 2718`
    - dev/editor: `marimo edit marimo_app.py ... --port 2719`
  - starts embedded Grist on port 8484 with Binder-safe defaults:
    - Grist is copied from `gristlabs/grist:1.7.14` into the image.
    - `PORT` is scoped only to the Grist subprocess so `jupyterhub-singleuser` keeps its Hub port contract.
    - persistent paths default under `${HOME}/grist-persist`; the Binder image treats them as runtime-local state.
  - MUST end with `exec "$@"` so the image’s default CMD is still executed (repo2docker/start-script semantics)
- `.binder/Dockerfile` (mirrored by root `Dockerfile` for local `docker build .` compatibility):
  - bakes Python deps into the image (from `requirements.txt`)
  - embeds the pinned Grist runtime via a multi-stage copy from `gristlabs/grist:1.7.14`
  - MUST explicitly wire `.binder/start` as the runtime entrypoint when a Dockerfile is present (repo2docker will not auto-wire it):
    - `ENTRYPOINT ["/home/jovyan/.binder/start"]`
    - `CMD ["jupyterhub-singleuser"]`
- `.jupyter/jupyter_server_config.py`:
  - enables `marimo_redirect` as a Jupyter Server extension
- `marimo_redirect.py`:
  - redirects `/` to `${JUPYTERHUB_SERVICE_PREFIX}proxy/2718/`
  - preserves `?token=...` when present (so BinderHub/JupyterHub deployments that rely on URL token auth don’t drop you onto `/login`)
- `scripts/docker-build.sh`:
  - local helper that builds the Docker image (matches CI semantics)
  - override tag via `IMAGE=...` (default: `marimo-binder:local`)
- `scripts/docker-smoke.sh`:
  - local helper that builds + runs the same smoke checks as CI (`docker-smoke`)
  - prints shipped versions + asserts `.binder/start` starts marimo and embedded Grist
  - suppresses `.binder/start` service logs on success and redacts Grist one-time boot keys from failure logs
- `scripts/grist-runtime-smoke.sh`:
  - lightweight Grist-only smoke gate; runs `gristlabs/grist:1.7.14` directly with the same defaults and verifies HTTP readiness on port 8484
  - avoids printing Grist boot keys in success logs and redacts the boot key in failure logs

## Decisions
- Decision: preserve `?token=...` on the root redirect (`/` → `proxy/2718/`).
  - Rationale: some BinderHub/JupyterHub deployments hand users a ready URL + token and rely on the singleuser server accepting URL token auth; dropping the token during redirect forces an unnecessary `/login` interstitial.
  - Implication: `marimo_redirect.py` forwards only the `token` query param; other query params are intentionally not propagated.

- Decision: keep `.binder/Dockerfile` as the canonical Binder Dockerfile and keep the root `Dockerfile` as an exact mirror for local compatibility.
  - Rationale: this repo is intended to be a Binder target; when `.binder/` exists, repo2docker discovers Dockerfile at `.binder/Dockerfile`, not at repo root.
  - Implication: CI/build helpers must use `docker build -f .binder/Dockerfile ...`; the Dockerfile must explicitly wire `.binder/start`.

- Decision: bake Python deps into the image (do not `pip install` at runtime).
  - Rationale: Binder/JupyterHub startup should be fast and not depend on network availability at container start.
  - Implication: `.binder/start` must remain network-free; add/update deps by editing `requirements.txt` + rebuilding the image.

- Decision: embed Grist from the pinned `gristlabs/grist:1.7.14` image instead of installing/building Grist from source in this repo.
  - Rationale: keeps the Binder image contract explicit while avoiding a second source-build toolchain inside the marimo repo.
  - Implication: Grist runtime updates are explicit Dockerfile pin changes plus smoke/contract test updates; the Grist process listens on port 8484 behind the existing JupyterHub proxy.

- Decision: add a lightweight Grist-only smoke gate separate from the full Binder image smoke.
  - Rationale: local full-image smoke can be dominated by heavyweight demo deps (`torch` / CUDA wheel downloads); Grist runtime correctness should still have a fast, focused gate.
  - Implication: use `scripts/grist-runtime-smoke.sh` for quick Grist runtime evidence and keep `scripts/docker-smoke.sh` / GitHub CI as the full-image gate.

- Decision: keep the default Binder image lightweight and split heavyweight embeddings deps into `requirements-heavy.txt`.
  - Rationale: repeated saucy.haus pinned-commit launches reached `Successfully built` / `Successfully tagged` and then timed out during Docker push/post-build, while local smoke was dominated by `torch` / CUDA-sized wheels; Grist/marimo default readiness should not depend on those heavy optional ML deps.
  - Implication: `requirements.txt` stays pinned and lightweight for default Binder/Grist smoke; `notebooks/algorithms/visualizing-embeddings.py` remains in-repo but requires explicit heavy-demo setup (`requirements-heavy.txt` plus `MARIMO_APP=notebooks/algorithms/visualizing-embeddings.py`).

- Decision: pin marimo deps in `requirements.txt`.
  - Rationale: Binder/repo2docker builds should be reproducible and not float to incompatible marimo / extension versions.
  - Implication: `requirements.txt` uses `==` pins; updating requires an explicit version bump + rebuild.

## Known limitations / sanity notes
- Local sanity note (verified): running `jupyterhub-singleuser` outside a real JupyterHub/Binder context can fail due to missing env (observed: `Missing required environment $JUPYTERHUB_SERVICE_URL`). This is expected for local `docker run` unless you inject the full Hub env.
- Local full-image smoke caveat (verified 2026-06-07 before heavy-dep split): `scripts/docker-smoke.sh` used to spend many minutes pulling heavyweight `torch` / CUDA wheels from `requirements.txt`; do not treat a locally killed old full-image smoke as evidence that the Grist sidecar failed. Current direction is to keep default smoke lightweight via `marimo_app.py` + `requirements.txt`, with heavy embeddings deps only in `requirements-heavy.txt`.
- Remote BinderHub caveat (verified on `binderhub.saucy.haus` for commit `5291f255` on 2026-06-07): after the Dockerfile UID-collision fix, the build reaches `Successfully built` / `Successfully tagged`, then BinderHub reports `UnixHTTPConnectionPool(host='localhost', port=None): Read timed out. (read timeout=60)`. Treat this as BinderHub/Docker post-build timeout pressure on a heavy image, not a marimo 500 or Grist runtime traceback, unless a later ready pod provides contradictory runtime logs.

## Dev/editor access (status: verified working)
- App mode is the default landing experience (port 2718).
- Dev/editor mode is a separate marimo instance (port 2719) and is reachable via:
  - `/dev` (Jupyter Server redirect), or
  - Binder `urlpath=proxy%2F2719%2F`.
- Embedded Grist is a separate sidecar process (port 8484) and is reachable via:
  - `/grist/` (Jupyter Server redirect), or
  - Binder `urlpath=proxy%2F8484%2F`.
- Note: there is no guarantee marimo’s in-app UI will expose a “switch to edit mode” button when running behind a Hub proxy; keeping the explicit `/dev` path is the supported UX.

## Local-vs-remote behavior (important)
- Remote (BinderHub/JupyterHub): users arrive authenticated via the Hub; desired behavior is: landing URL → marimo UI without any additional token prompts.
- Local (`repo2docker` only): there is no Hub, so Jupyter Server will typically require a token/login before it will proxy to marimo. This is expected.
## Verified local behavior (repo2docker):
  - `GET /` → `302 Location: /proxy/2718/`
  - `GET /proxy/2718/` (unauthenticated) → `302 Location: /login?next=...`
  - `GET /proxy/2718/?token=<jupyter token>` → `200` and marimo HTML is served.

## Binder launch URLs (opens directly into app/dev/Grist)
- BinderHub supports a `urlpath=...` parameter that is appended under the runtime user prefix.
- App (default):
  - `...?urlpath=proxy%2F2718%2F`
- Dev/editor:
  - `...?urlpath=proxy%2F2719%2F`
- Grist:
  - `...?urlpath=proxy%2F8484%2F`

Important: the BinderHub URL shape is `<ORG>/<REPO>/<REF>`.
- Correct (saucy.haus): `https://binderhub.saucy.haus/v2/gh/NativeVex/marimo-binder/<REF>`
- Incorrect (will fail to resolve on saucy.haus): `.../v2/gh/git@github.com:NativeVex/marimo-binder.git/<REF>`

Recommendation: for “remote-faithful” local testing without BinderHub, run a local JupyterHub (DummyAuthenticator) that spawns this image as the singleuser server; this preserves Hub cookies + service prefix semantics.
