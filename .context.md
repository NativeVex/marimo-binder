# marimo-binder — context

## Purpose / North Star
A minimal Binder / JupyterHub-compatible repo that launches JupyterLab and auto-starts a marimo app (`notebook.py`), with the *entry URL* landing on marimo (via Jupyter Server root redirect to the JupyterHub proxy URL).

## Non-goals
- Not a general-purpose JupyterHub deployment repo.
- Not optimizing for complex multi-user auth, persistence, or production hardening.

## Current direction (high level)
- Keep the setup Binder-friendly and reproducible.
- Prefer repo2docker-supported config files; only use a custom `Dockerfile` when required.

## Constraints / gotchas (verified)
- repo2docker semantics: If a `Dockerfile` is present, *all other configuration files are ignored* and a regular Docker build is performed. (repo2docker docs: “Dockerfile - Advanced environments”)
- This repo currently contains a `Dockerfile`, so Binder/repo2docker will ignore other config mechanisms.
- Git workflow: use GitButler (`but`) for mutations (staging/commits/branches).

## Repo entrypoints / behavior
- `.binder/start`:
  - installs `requirements.txt` at runtime
  - starts marimo on port 2718
  - MUST end with `exec "$@"` so the image’s default CMD is still executed (repo2docker/start-script semantics)
  - NOTE: If a `Dockerfile` is used, you must explicitly wire this script as the runtime entrypoint; it is not automatically honored just because it exists under `.binder/`.
- `.jupyter/jupyter_server_config.py`:
  - enables `marimo_redirect` as a Jupyter Server extension
  - prepends repo dir to `sys.path` so `marimo_redirect.py` is importable
- `marimo_redirect.py`:
  - redirects `/` to `${JUPYTERHUB_SERVICE_PREFIX}proxy/2718/`
